# 综述审稿助手需求规格说明

## 1. 文档目的

本文定义“结构化规则引擎 + AI 语义审稿”路线的 MVP 需求与验收口径。需求中的“必须”表示上线门槛；“应当”表示优先实现；“可选”表示后续能力。

## 2. 业务对象与状态

| 对象 | 关键字段 | 生命周期 |
| --- | --- | --- |
| 期刊 | 名称、语言、默认模板、成员 | 启用/停用 |
| 模板 | 期刊、文章类型、规则集版本、状态 | 草稿 -> 待发布 -> 已发布 -> 已归档 |
| 规则 | 维度、类型、条件、严重级别、双语文案、来源 | 草稿/启用/停用 |
| 稿件 | 原文件、解析文本、结构、附件、哈希 | 上传 -> 解析中 -> 可审稿 -> 已归档 |
| 审稿任务 | 稿件、模板版本、报告语言、审稿人 | 待开始 -> 运行中 -> 待复核 -> 已定稿 |
| 审稿发现 | 规则、状态、证据、理由、置信度、最终处理 | AI 草稿 -> 审稿人确认 -> 已定稿 |

## 3. 功能需求

### FR-01 规则清单导入

1. 编辑必须能够上传 `.docx`、`.pdf`、`.xlsx`、`.xls` 格式的清单。
2. 系统必须保存原始文件、文件哈希、上传者和上传时间。
3. 系统必须抽取标题、表格、编号、层级、单元格和 PDF 页码，并生成候选规则。
4. 无法可靠提取的扫描 PDF 必须先经过 OCR，并标注 OCR 置信度。
5. 系统不得自动将候选规则直接用于审稿；编辑发布前必须完成确认。

验收：导入当前的 `综述清单.pdf` 和 `文章结构.docx` 后，可在界面上看到可编辑的 Narrative Review 候选规则及其来源位置。

### FR-02 模板和规则管理

1. 编辑必须能创建期刊、文章类型和模板；文章类型至少支持 Narrative Review。
2. 编辑必须能新增、编辑、拆分、合并、排序、禁用和删除未发布规则。
3. 每条规则必须含：唯一标识、维度、标题、规则说明、规则类型、严重级别、是否必填、适用条件、报告中英文文案和来源引用。
4. 模板发布后不得原地修改；修改必须产生新版本。
5. 历史审稿任务必须永久绑定其发起时使用的模板版本。

验收：编辑将“方法”中一条复合规则拆分为多条后，发布新版本；新任务使用新版本，旧任务仍显示旧版本。

### FR-03 稿件上传与解析

1. 审稿人必须能上传 `.docx` 或 `.pdf` 稿件；系统应限制文件大小、页数和恶意文件类型。
2. DOCX 解析必须保留标题、段落、表格、图片、脚注、参考文献和样式层级。
3. PDF 解析必须保留页码、文本块、表格候选、图片区域；扫描 PDF 需 OCR。
4. 系统必须构建统一稿件模型，任何引用证据至少可定位至页码或章节+段落；DOCX 优先定位至段落和字符范围。
5. 解析失败必须给出可理解的原因和重试入口，不能静默生成不完整审稿。

验收：同一篇 DOCX 和导出的 PDF 分别上传后，系统都能识别标题、摘要、引言、方法、主体、结论、参考文献、图表及其对应位置。

### FR-04 自动审稿

1. 审稿人必须选择期刊模板版本和输出语言后才可启动任务。
2. 系统必须区分硬规则、结构规则、语义规则、交叉一致性规则和人工/外部核验规则。
3. 每条适用规则必须返回：到位、部分到位、未到位、无法判断、无需适用之一。
4. “未到位”和“部分到位”必须给出具体缺失说明和可操作的修改建议。
5. 语义判断必须只以稿件和已配置的受信资料为依据；无证据时应返回“无法判断”或降低置信度。
6. 系统必须在运行中显示阶段状态（解析、结构检查、内容审稿、生成报告），失败可重试。

验收：对缺少精确检索日期、检索时间范围、纳排标准和筛选过程的测试稿，系统分别给出对应发现，并定位至方法部分或说明未找到。

### FR-05 审稿工作台与人工复核

1. 审稿人必须可按维度、状态、严重级别和置信度筛选发现。
2. 点击发现时，系统必须并排显示规则、AI 判断、原文证据和稿件上下文。
3. 审稿人必须能接受、编辑后接受、驳回、标记待讨论并填写备注。
4. 审稿人编辑后的内容不得被后续模型运行覆盖；再次运行必须创建新一轮 AI 草稿。
5. 系统必须记录操作者、时间、原值、新值和理由。

验收：审稿人将 AI 的“未到位”改为“部分到位”并补充意见后，导出报告仅使用该最终结论，审计日志仍保留 AI 初始判断。

### FR-06 双语报告与导出

1. 审稿人必须可在任务开始前或导出前选择中文或英文报告。
2. 系统必须支持总览、结构审稿、内容审稿、待人工核验项和审稿人附言。
3. 报告中的每项必须显示最终状态；可配置是否显示 AI 置信度和证据摘录。
4. 导出格式 MVP 至少支持 PDF 和 DOCX；可选支持 HTML、JSON。
5. 报告必须标注模板版本、生成时间和“AI 初审经人工复核”的状态。

## 4. Narrative Review 首期规则要求

### 4.1 结构层

- 标题页、标题、作者、单位、通讯信息、Running title、词数、贡献信息是否存在；
- 摘要长度是否在 200–350 词，关键词是否为 3–5 个；
- 引言是否具备 Background、Rationale and knowledge gap、Objective；
- 主体不以 “Main body” 作为下级标题，并包含综述优势与局限讨论；
- 是否有结论、参考文献、图表/图例及适用时的致谢、资金、利益冲突、伦理与清单声明；
- Narrative Review 总字数是否小于 6000（以期刊模板配置为准）。

### 4.2 内容层

- 标题是否突出重点、覆盖主题、明确 Narrative Review；更新类是否标注 updated；
- 摘要是否呈现 Background and Objective、Methods、Key Content and Findings、Conclusions；
- 方法是否报告精确检索日期、精确起止时间、数据库、搜索词/策略、纳排标准、筛选流程；
- 正文是否概述文献结果、评价纳入文献质量或局限、说明综述自身优势/局限、提出未来研究、合理使用总结图表；
- 结论是否说明对临床实践、政策或未来研究的意义，并被正文支撑；
- 参考文献是否有缺引、冗余引用、时效性不足、列表与正文不一致的可疑情况；
- 图表是否被正文引用、说明完整、数据一致；版权许可和视觉质量需标记为人工核验。

## 5. 非功能需求

| 编号 | 要求 |
| --- | --- |
| NFR-01 安全 | 全链路 TLS、静态加密、按期刊隔离访问；上传文件必须做恶意文件扫描和类型校验。 |
| NFR-02 隐私 | 未公开稿件不得被用于模型训练；支持按期刊设置保留期和删除流程。 |
| NFR-03 可审计 | 模板、模型版本、提示词版本、输入摘要、证据、人工改动和导出均留痕。 |
| NFR-04 可解释 | 每个确定结论必须有规则依据和稿件内证据；无证据不得给高置信度结论。 |
| NFR-05 性能 | 常规稿件应在目标服务等级内完成；长文档采用异步队列、进度和失败重试。 |
| NFR-06 可用性 | 中文和英文操作界面与报告可切换；核心审稿工作台支持桌面端。 |
| NFR-07 质量 | 硬规则必须有自动化测试；语义规则必须有标注集评估和人工抽检。 |

## 6. 风险与处理规则

| 风险 | 系统处理 |
| --- | --- |
| OCR 误识别 | 展示 OCR 置信度；低置信度章节/证据提示人工检查。 |
| LLM 幻觉 | 强制引用受控证据；无证据降级为无法判断；保存推理摘要而非让模型自由断言。 |
| 规则歧义 | 转换为候选规则并要求编辑发布确认；不从模糊文字直接执行。 |
| 图表/版权不可验证 | 输出“需人工/外部核验”，不做虚假的通过判断。 |
| 双语语义漂移 | 规则以统一语义字段为真源；中英文仅为展示文案，并经过编辑校对。 |
