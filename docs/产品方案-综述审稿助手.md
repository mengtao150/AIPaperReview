# 综述审稿助手产品方案

## 1. 产品定位

“综述审稿助手”是一套面向期刊编辑和审稿人的 AI 初审系统。它依据可配置、可版本化的期刊清单，对 Word 或 PDF 稿件完成两层检查：

1. 结构审稿：章节、元信息、字数、图表、引文等是否满足模板和规则；
2. 内容审稿：稿件是否实际呈现规则要求的内容，例如研究背景、rationale、knowledge gap、检索策略、综述局限性和临床/政策/研究意义。

系统自动给出初审结论；审稿人逐条接受、修改或驳回后，形成正式审稿报告。产品不是自动拒稿器：最终决定权始终在审稿人。

## 2. 首期目标与范围

首期服务对象是 AME 风格的 Narrative Review（叙述性综述），但产品底层不将 AME 或 Narrative Review 写死。编辑可以上传 Word、PDF、Excel 清单，将其转换为可编辑的规则模板，后续扩展至 Mini Review、Clinical Practice Review 和其他期刊。

首期支持：

- 稿件上传：`.docx`、文本型 PDF 和扫描型 PDF；
- 规则清单上传：`.docx`、`.pdf`、`.xlsx`、`.xls`；
- 中英文界面与中英文报告任选；
- 逐项审稿、原文证据定位、置信度、人工复核、报告导出；
- 模板、规则和审稿结论的全程版本追溯。

首期不包含：投稿系统替代、同行评审专家匹配、版面编排、自动决定录用/拒稿、跨出版社版权许可核验的最终法律结论。

## 3. 用户与职责

| 用户 | 核心任务 | 权限边界 |
| --- | --- | --- |
| 期刊编辑 | 导入、编辑、发布审核模板；查看审稿统计 | 管理本期刊模板和人员，不改写审稿人的最终意见 |
| 审稿人 | 上传或领取稿件；发起自动审稿；复核结论；导出报告 | 对被分配稿件作最终确认 |
| 期刊管理员 | 配置期刊、成员、默认模板、留存策略 | 不参与具体规则判断 |
| 系统管理员 | 运营、模型和安全审计 | 无权随意查看稿件正文；访问需留痕 |

## 4. 核心体验

```mermaid
flowchart LR
  A[编辑上传审核清单] --> B[AI 转为候选规则]
  B --> C[编辑确认并发布模板版本]
  D[审稿人上传 DOCX/PDF] --> E[选择期刊与模板]
  C --> E
  E --> F[自动解析与逐项审稿]
  F --> G[证据化初审报告]
  G --> H[审稿人接受、修改或驳回]
  H --> I[导出正式审稿意见]
```

### 4.1 编辑：将清单变成规则模板

编辑上传清单后，系统展示“候选规则草稿”，而不是直接上线。每条候选规则包含：所属文章类型、审核维度、规则名称、规则说明、规则类型、严重级别、判定方式、报告中英文文案和来源页码/单元格。编辑可拆分、合并、禁用、设为必填并发布一个版本。

例如，当前 Narrative Review 清单中的“检索日期是否具体到年月日”应成为独立的必填内容规则；“检索词和检索策略”可拆成“覆盖范围”和“AND/OR/NOT 或补充表呈现”两条规则；“主要内容”下的多个要求也应拆成可独立判定的规则。

### 4.2 审稿人：自动初审后做专业判断

审稿人选择模板版本与报告语言后发起审稿。系统为每条规则返回：

- 状态：到位、部分到位、未到位、无法判断、无需适用；
- 结论摘要及明确的修改建议；
- 支持或反驳结论的原文摘录；
- 证据位置：章节、段落、页码、表格/图号；
- 置信度：高、中、低；
- 规则来源和版本；
- 审稿人的最终处理状态：接受、编辑后接受、驳回、标记待讨论。

低置信度与“无法判断”的结果默认进入“重点复核”队列，不能伪装成确定性缺失。

### 4.3 报告交付

正式报告分为三个阅读层次：

1. 总览：适用模板、总体风险、到位率、按严重级别的缺失数；
2. 结构审稿：按稿件结构列出缺失项与定位；
3. 内容审稿：按清单逐项列出是否呈现、证据和建议。

报告可选择中文或英文；语言改变的是报告表述，不改变引用的稿件原文和结论记录。

## 5. MVP 模块

| 模块 | MVP 能力 | 价值 |
| --- | --- | --- |
| 规则模板中心 | 导入、AI 抽取、编辑、发布、回滚、版本对比 | 让规则从文档资产变成可执行资产 |
| 稿件解析中心 | DOCX/PDF 解析、OCR、章节识别、页码/段落定位 | 为所有结论提供可回查证据 |
| 审稿引擎 | 硬规则校验、语义规则评估、证据检索、置信度 | 自动形成可靠的初审底稿 |
| 审稿工作台 | 筛选、原文对照、逐条修改/接受/驳回 | 保留审稿人的专业控制权 |
| 报告中心 | 中英文报告、导出、审计记录 | 交付可使用、可追溯的意见 |
| 管理与审计 | 角色权限、访问日志、保留/删除策略 | 适合处理未公开稿件 |

## 6. Narrative Review 基线模板

根据当前两份规则源文件，首套模板至少覆盖：

- 标题：主题聚焦、覆盖完整、Narrative Review 定位；更新类综述标示 updated，建议含检索时间范围；
- 摘要：Background and Objective、Methods、Key Content and Findings、Conclusions 四项是否呈现；
- 关键词：3–5 个，且与主题密切相关；
- 引言：背景、rationale 与 knowledge gap、明确 objective、遵循清单声明；
- 方法：检索日期和时间范围精确到年月日，数据库、检索词/布尔策略、纳排标准、筛选过程；
- 主体：概述文献结果，评价文献局限性/质量，说明本综述优势与局限，提出未来研究并总结图表；
- 结论：说明对临床实践、政策或未来研究的意义，且有正文证据支撑；
- 通用质量：字数、引用时效性与完整性、图表完整性/一致性/版权、缩写一致性、伦理和格式项。

其中“字数上限”“关键词数量”“运行标题长度”等属于硬规则；“主题是否聚焦”“结论是否有证据支撑”“综述是否有深度”等属于语义规则，必须显示理由和证据。

## 7. 关键产品原则

- 证据优先：没有稿件内证据的 AI 结论，不能作为确定结论输出。
- 规则可解释：每个发现可回到模板版本、规则原文、稿件位置和模型理由。
- 人在回路：AI 提供初审，审稿人确定最终报告；审稿人的修订会被保留但不自动改写期刊规则。
- 模板优先：不同期刊、不同综述类型通过模板解决，而不是通过复制代码解决。
- 谨慎陈述：对“图像清晰度”“版权已获许可”“来源文献真实准确”等无法仅凭正文确认的项目，应输出“需要人工/外部核验”，而非误判为通过或不通过。

## 8. 分期路线图

| 阶段 | 范围 | 完成标志 |
| --- | --- | --- |
| P0 规则数字化 | Narrative Review 模板、规则导入草稿、编辑发布 | 当前清单可被稳定转为可编辑模板 |
| P1 自动初审 MVP | DOCX/PDF、结构/内容审稿、证据、人工复核、双语报告 | 审稿人能用一套模板完成一次闭环审稿 |
| P2 质量增强 | 引文核验、图表/图片质量、规则学习、模板分析 | 复杂规则的覆盖率和准确率可衡量提升 |
| P3 期刊集成 | 投稿系统接口、多人协作、运营仪表盘 | 可嵌入编辑部日常流程 |

## 9. 成功指标

- 编辑确认后，清单规则抽取的字段准确率不低于 90%；
- 对明确结构项的召回率不低于 95%；
- 高置信度“未到位”结论被审稿人接受或仅轻微修改的比例不低于 85%；
- 单篇常规综述的自动初审在可接受的排队时间内完成；
- 每条正式报告结论均能回链至规则版本和稿件证据。
